# Indexer DB Optimization Plan

## Context

All binary data (addresses, payloads, hashes) is currently stored as `character varying` in PostgreSQL. This wastes storage and is slower to compare than native binary types. The `vara-eth/indexer` already uses `bytea` correctly as the reference implementation. Additionally, the message tables need 3 new fields for Sails IDL v2 parsing (`isSailsIdlV2`, `header`, `routeIdx`).

Migrations are **auto-generated by TypeORM** after entity changes.

---

## Key Design Decision: TypeORM Column Transformers

Use column transformers so TypeScript types stay as `string` — no changes needed in the squid writer or explorer query code:

```typescript
const hexToBytea = {
  to: (v: string | null) => (v ? Buffer.from(v.slice(2), 'hex') : null),
  from: (v: Buffer | null) => (v ? '0x' + v.toString('hex') : null),
};

@Column({ type: 'bytea', transformer: hexToBytea })
public destination: string;
```

## Phase 1 — bytea conversion + new fields + indexes

### `src/entities/base.entity.ts`
- `blockHash: varchar(66)` → `bytea` (with `hexToBytea` transformer)
- `blockNumber: varchar` → `bigint` (transformer keeps TS type as `string`, stored as `int8` for correct numeric ordering)

### `src/entities/message-to-program.entity.ts`
Convert to bytea: `destination`, `source`, `payload`, `replyToMessageId`

New fields:
- `isSailsIdlV2: boolean` — `@Column({ name: 'is_sails_idl_v2', default: false })`
- `header: Buffer | null` — `@Column({ type: 'bytea', nullable: true })`
- `routeIdx: Buffer | null` — `@Column({ type: 'bytea', nullable: true, name: 'route_idx' })`

Replace `@Index()` on `destination` and `source` with composite indexes (see Indexes section).

### `src/entities/message-from-program.entity.ts`
Convert to bytea: `destination`, `source`, `payload`, `parentId`, `replyToMessageId`

Same 3 new fields as above.

Replace `@Index()` on `destination` and `source` with composite indexes (see Indexes section).

### `src/entities/program.entity.ts`
Convert to bytea: `owner`, `codeId`

### `src/entities/code.entity.ts`
Convert to bytea: `uploadedBy`

### `src/entities/event.entity.ts`
Convert to bytea: `source`, `payload`, `parentId`

### `src/entities/voucher.entity.ts`
Convert to bytea: `owner`, `spender`

## Phase 2 — `id` fields to bytea (deferred)

Converting `id` (primary key) to `bytea` requires verifying how `@subsquid/typeorm-store` handles bytea PKs — specifically whether `store.findOneBy`, `store.save`, and `store.remove` work correctly when the PK is a `Buffer` rather than a string. Once confirmed, apply the same `hexToBytea` transformer to `id` on all entities.

## Indexes

### Composite indexes on `(destination, timestamp)` and `(source, timestamp)`
Replaces the existing single-column indexes on both message tables. Eliminates the sort step on the most common query pattern (`WHERE destination = $1 ORDER BY timestamp DESC`).

```typescript
@Entity({ name: 'message_to_program' })
@Index(['destination', 'timestamp'])
@Index(['source', 'timestamp'])
export class MessageToProgram extends BaseEntity { ... }
```

### BRIN index on `timestamp`
Applied to all `BaseEntity` tables: `message_to_program`, `message_from_program`, `program`, `code`, `event`. Near-zero write overhead; ~50 KB vs ~200 MB for a B-tree on 10M rows. Useful for `from`/`to` time-range queries with no other filter.

TypeORM's `IndexOptions` type does not include `using`, so BRIN indexes cannot be declared via decorators. Add them **manually** to the generated migration's `up` method:

```sql
CREATE INDEX CONCURRENTLY ON message_to_program USING brin (timestamp);
CREATE INDEX CONCURRENTLY ON message_from_program USING brin (timestamp);
CREATE INDEX CONCURRENTLY ON program USING brin (timestamp);
CREATE INDEX CONCURRENTLY ON code USING brin (timestamp);
CREATE INDEX CONCURRENTLY ON event USING brin (timestamp);
```

### Partial index on `parent_id`
Only on `message_from_program`. Most messages have no parent; indexing only non-null rows keeps the index small.

```typescript
@Index(['parentId'], { where: '"parent_id" IS NOT NULL' })
```

### Not indexed
- `id` (PK): auto-indexed by PostgreSQL, no additional index needed
- `replyToMessageId`: never used as a WHERE filter, index would be write-only overhead
- `expiration`: `isInMailbox` is not a common query

## Squid: New field population

In `idea/gear/squid/src/temp-state.ts`, update `addMsgToProgram` and `addMsgFromProgram` to populate the new fields using `SailsMessageHeader.tryFromBytes` (already imported). `SailsMessageHeader` API:

```typescript
static tryFromBytes(bytes: Uint8Array): { ok: boolean; header: SailsMessageHeader | undefined }
// header fields: version, hlen, interfaceId, routeIdx (number), entryId
toBytes(): Uint8Array
```

For storage, `header` should be the full serialized header bytes (`header.toBytes()`), and `routeIdx` should be the route index as a 4-byte little-endian buffer.

## Performance Impact Summary

| Change | Writes | Reads |
|---|---|---|
| bytea on binary columns | Slight improvement (smaller rows/indexes) | Moderate improvement (smaller B-tree indexes, faster equality comparison) |
| blockNumber as bigint | Negligible | Correctness fix (numeric sort) |
| Composite indexes (replaces single-column) | Neutral (same count) | **Largest win** — eliminates sort step on paginated list queries |
| BRIN on timestamp | Near zero | Moderate for time-range-only queries |
| Partial index on parent_id | Very low | Minor improvement for child-message lookups |
